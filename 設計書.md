Pokédex Online Next Architecture Specification
Overview

This document describes the architecture for the next-generation Pokédex Online web application.

The goals are:

Use Nuxt (latest version) for frontend

Deploy as fully static site

Work on Apache without Node runtime

Use the pokedex repository as the data source

Do not modify source JSON files

Optimize data delivery for large JSON datasets

The application will use a build-time data pipeline that transforms the data source into optimized web assets.

Repository Structure

Two repositories will remain separate.

towakey
 ├ pokedex
 │
 └ pokedex-online
pokedex (data source)

This repository contains the authoritative dataset.

It includes files such as:

pokedex.json
kanto.json
johto.json
paldea.json
...

These files must never be modified by the web application.

They are treated as read-only source data.

The dataset includes Pokémon data such as:

names (multi-language)

forms

stats

classification

height / weight

dex descriptions

Data Processing Pipeline

Large JSON files must not be loaded directly by the browser.

Instead, create optimized web data during build.

Pipeline:

pokedex repo
      ↓
build-data script
      ↓
generated-data
      ↓
Nuxt static site

Pokemon Data Files

Each Pokémon must have an individual JSON file.

Example:

generated-data/pokemon/25.json

Example content:

{
  "id": 25,
  "name": "ピカチュウ",
  "types": ["electric"],
  "height": 0.4,
  "weight": 6.0
}

This allows loading only necessary data.

Index Data

The Pokédex list page should use a lightweight index.

generated-data/index.json

Example:

[
  {
    "id": 1,
    "name": "フシギダネ",
    "types": ["grass","poison"]
  }
]

This avoids loading full Pokémon data.

Region Dex Files

Regional Pokédex files must be generated.

Example:

generated-data/region/kanto.json

Structure:

[
  {
    "dex": 25,
    "pokemon_id": 25
  }
]

The frontend will resolve the Pokémon data using pokemon_id.

Search Index

A search-optimized dataset must be generated.

generated-data/search-index.json

Example:

[
  {
    "id": 25,
    "name": "ピカチュウ",
    "types": ["electric"]
  }
]

Used for client-side search.

Design Principles
Data source isolation

The pokedex repository is authoritative data.

The web application must never modify source files.

Build-time optimization

All heavy processing happens during build.

Browser loads only required data.

Static-first architecture

The system must work on:

Apache

GitHub Pages

CDN

No runtime backend required.

Future Extensions

The architecture should allow:

additional regional dex

multilingual search

progressive loading

PWA support

offline Pokédex
